> TL,DR：  
> 浏览器自动化正在分化成三条路线：一条是 Playwright / Selenium / Puppeteer 这类“开发者控制浏览器”的传统框架；一条是 agent-browser / browser-use / Pinchtab / chrome-agent-skill 这类“让 AI 更容易理解和操作网页”的 Agent-first 工具；还有一条是 Browser Cluster 这类“把浏览器自动化做成基础设施”的系统化方案。真正的选型关键，不是哪个工具功能最多，而是你到底是在写脚本、养 Agent，还是搭系统。

本文通过调研各种工具使用的文章和测评，总结了一下表格供大家参阅，更多详细细节请看benren其他文章，初出茅庐，请大家多多关注，如果不足还望不吝赐教

### <span id="_6"></span>一眼选型版总结表

| 你的目标 | 更推荐的方案 | 原因 |
|----|----|----|
| 传统自动化测试 | **Playwright / Selenium** | 控制力强、测试生态成熟；但不是专门为 LLM 设计的 |
| 动态网页抓取、登录后抓取 | **CDP + Puppeteer** 或 **OpenClaw 组合路线** | 能直接连浏览器、抓动态内容、执行 JS、处理登录后页面 |
| AI 自动点网页，且想省 token | **agent-browser** | A11y snapshot + ref，明显减少 DOM 噪音和上下文消耗 |
| 复杂链路、复杂表单、依赖登录态 | **browser-use** | 你材料里的实测倾向是：复杂交互与登录态复用更省事、更稳 |
| 直接操作我正在用的 Chrome | **chrome-agent-skill** | 真正复用当前浏览器和登录态，过程可见、调试方便 |
| 想要轻量、HTTP 化、跨语言调用 | **Pinchtab** | 单二进制、REST API、A11y-first、会话持久化，轻巧得像把瑞士小刀 |
| 想做大规模并发采集 / 持续任务系统 | **Browser Cluster** | 它是系统，不是玩具；核心价值在分布式、缓存、代理、可视化和调度\< |
